## Learnings

- Hook names in source differ from spec: `onMessage` → `message_received`; `onToolCall` → `before_tool_call` / `after_tool_call`
- LoRA adapters require GGUF conversion before llama.cpp load — `model.save_lora()` outputs safetensors only; `convert-lora-to-gguf.py` is a required pipeline step
- `--lora-init-without-apply` startup flag required for hot-swap; per-request `lora` override preferred over global `POST /lora-adapters` for concurrent use
- DPO `reward` field (`sentiment × magnitude`) is not consumed by the trainer — use as candidate filter gate and held-out stratification only; reserve for reward-weighted DPO variants (SimPO, Cal-DPO) later
- Oracle constraint: DPO does not theoretically require a stronger oracle — reference model is typically the same frozen SFT checkpoint. Constraint is a quality-ceiling argument: oracle capability sets the ceiling for improvement. Weaker oracle → low-contrast pairs → marginal gains or degradation
- Attribution across session resets is recoverable: `origin` field survives reset; archived transcripts at `{sessionFile}.reset.{ISO-timestamp}` are not deleted
- Subagent spawn method not found in source — `spawnedBy` field confirms the mechanism exists but endpoint name/params unverified. Must resolve before Phase 2

## Risks

- Subagent spawn API unverified — if the gateway method differs from expected, Phases 2 implementation of Feedback Analyzer and Candidate Synthesizer invocation may need significant rework
- LoRA GGUF conversion compatibility — `convert-lora-to-gguf.py` behavior may vary by model architecture; untested on the specific target model
- llama.cpp hot-swap concurrency caveat — global `POST /lora-adapters` is unsafe under concurrent slots; per-request override mitigates but requires careful request routing
- Known bug: llama.cpp aborts when loading multiple LoRA adapters in some versions (issue #18466) — pin version or test before training produces second adapter generation

## Strategy

Three implementation phases defined:
- **Phase 1 (Feedback Capture & Attribution):** hooks → structured events → attributed turns. Foundation everything else depends on.
- **Phase 2 (Candidate Pipeline):** attributed events → confirmed DPO candidates in buffer. Subagent invocations; operator confirmation loop.
- **Phase 3 (Training & Deployment):** buffer → Unsloth DPO → GGUF conversion → deployment gate → llama.cpp hot-swap.

Phase 1 begins after subagent spawn method verified (OQ #4).

## Decisions

- DPO for Phase 1; GRPO deferred
- Explicit operator positive signal only in v1 — no unattended buffer writes
- Deployment path: LoRA adapter (not full GGUF) via `POST /lora-adapters`
- `reward` field used as candidate filter gate, not training input

## Alignment

- TDD throughout all implementation phases: test task precedes every implementation task
- Phase gate holds: no Phase 2 code before Phase 1 verification passes
- WaLRuS at each phase completion

## Tradeoffs

- DPO over GRPO: simpler pipeline, oracle at collection time, no live training dependencies — cost: can't use continuous reward signal in trainer
- Explicit positive signal only: safe, no buffer pollution — cost: lower throughput of positive candidates vs unattended oracle path
- LoRA over full GGUF deploy: fast iteration, no model reload — cost: conversion step in pipeline; GGUF format requirement adds a build artifact

## Alternatives

- GRPO: rejected for v1 — requires live oracle as reward function at training time, vLLM, heavier infrastructure
- Passive no-correction window for positive signal: rejected — "no correction ≠ success"; weak signal; buffer pollution risk confirmed by literature
- Full GGUF model swap on deploy: rejected — no model reload needed; LoRA hot-swap is sufficient and faster
