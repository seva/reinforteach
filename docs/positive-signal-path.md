# Discovery: Positive Signal Path — Options and Tradeoffs

## The problem

The spec says: "successful sessions without correction → oracle rates quality → positive candidates added without operator intervention."

Two things are undefined:
1. What is a "successful session"?
2. Should positive candidates bypass operator confirmation?

---

## What the hook layer gives us

`message_received` fires for every inbound message. There is no built-in "session closed" or "session succeeded" event. A session without correction means: no `message_received` event in the session window carried negative feedback sentiment. This is a weak signal — absence of correction ≠ success.

---

## Options

### Option 1 — No-correction window (passive)

**Mechanism:** After N turns with no `message_received` matching negative feedback patterns, assume success. Oracle rates the session transcript. If quality score ≥ `confidence_threshold`, auto-add positive candidate to buffer.

**For DPO:** Agent output = `chosen`. Oracle generates a degraded version = `rejected`.

**Tradeoffs:**
- No operator friction — fully unattended
- High false positive risk: operator may simply not be watching
- Oracle cost per session (one quality rating call per N-turn window)
- A degraded `rejected` generated by oracle may be low-quality synthetic data
- Silent buffer pollution is the failure mode — hard to detect

**Verdict:** Risky as primary path. Viable as a low-weight supplement if oracle confidence threshold is set high and volume is monitored.

---

### Option 2 — Oracle quality gate only (unattended)

**Mechanism:** Periodic cron reviews recently-completed session transcripts. Oracle scores quality. High-confidence sessions (above threshold) produce positive candidates automatically, no operator loop.

**Tradeoffs:**
- Higher throughput of positive candidates than Option 3
- Still unattended — carries buffer pollution risk
- Oracle must be run on every session transcript (cost scales with agent activity)
- "Recently completed" is not a native OpenClaw event; requires polling or a session-close hook

**Verdict:** Better than Option 1 (oracle-gated, not just absence-of-correction), but still unattended. Appropriate as a later-phase addition once the pipeline is validated.

---

### Option 3 — Explicit operator positive signal (recommended for v1)

**Mechanism:** Operator sends an explicit positive signal (e.g., "good", "correct", thumbs-up reaction). `message_received` hook detects positive sentiment. Same pipeline as negative feedback:

```
message_received (positive sentiment)
  → feedback analyzer: confirms positive, attributes to session turn
  → candidate synthesizer:
      prompt   = context that produced the attributed turn
      chosen   = what the agent actually produced  ← inverted from negative path
      rejected = oracle-generated degraded version
      reward   = sentiment × magnitude (positive)
  → confirmation handler: operator confirms the pair
  → training buffer
```

**Tradeoffs:**
- Operator remains in the loop — no unattended buffer writes
- Reuses the entire existing pipeline with minimal branching (only chosen/rejected are swapped)
- Lower throughput: depends on operator actively signaling
- Oracle-generated `rejected` is synthetic — lower data quality than a real mistake, but better than no negative example
- Operator burden is low: a single word or reaction suffices

**Verdict:** Recommended for v1. Safest design. No new infrastructure. Operator confirmation is the one consistent gate across all candidates.

---

## Decision recommendation

**Use Option 3 for Phase 1.** Rationale:

- Reuses the existing confirmation pipeline with minimal change — `chosen` and `rejected` are swapped, everything else is identical
- No unattended buffer writes in v1
- The `reward` field (sentiment × magnitude on positive) is naturally positive and provides balanced training signal alongside negative candidates
- Oracle-generated synthetic `rejected` is acceptable given that real negative examples from correction will dominate the buffer early on

**Option 2 can be added in a later phase** as a throughput supplement, once the pipeline is validated and the buffer has enough real signal to calibrate the oracle quality threshold against.

---

## Impact on architecture

Open Question #2 is resolved: "successful session" is not a passive no-correction window. It is an **explicit positive operator signal**, processed through the same pipeline with inverted chosen/rejected roles.

No unattended path in v1. The architecture constraint "operator confirmation required for all candidates" holds for both signal polarities.
